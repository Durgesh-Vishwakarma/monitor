### BUG-C1 Â· **Duplicate `ImageReader` + camera session built on wrong reader**
**File:** `MicService.kt` line 2180â€“2319  
**Problem:** `captureJpegOnce()` creates an `ImageReader` at line 2180 **with correct `size`**, then creates a **second shadow `ImageReader` at line 2224 with a hard-coded `1024Ã—1024`**. ### BUG-C2 Â· **`startAudioCapture()` is NOT coroutine-safe â€” double-start race**
**File:** `MicService.kt` line 2704â€“2705  
**Problem:** The guard `if (isCapturing) return` is a plain `@Volatile` read â€” **not atomic**. Two concurrent callers (e.g. `onWsOpen` + watchdog recovery) can both pass the guard, create two `AudioRecord` instances, and stomp each other. The second `AudioRecord` consumes the audio session but its reference is immediately overwritten, causing a **leaked AudioRecord** that holds the microphone hardware indefinitely. All subsequent mic-start attempts fail with `ERROR_BAD_VALUE` or silence.

**Fix:** Replace with `compareAndSet` on an `AtomicBoolean`, same as `isWsConnecting`.

### BUG-C3 Â· **`onWsDisconnected` calls `stopAudioCapture` which is synchronous, but the running coroutine is still alive**
**File:** `MicService.kt` line 676â€“684  
**Problem:** `stopAudioCapture()` sets `isCapturing = false` and calls `audioRecord?.stop()`. But the audio loop running on `Dispatchers.IO` is not cancelled â€” it continues one more iteration, calls `audioRecord?.read()` on the already-stopped AudioRecord (which returns `ERROR_INVALID_OPERATION`), then tries `safeSend` again, which triggers another `onWsDisconnected`, starting an infinite reconnect storm.

### BUG-C4 Â· **Heartbeat `syncCommandsNow()` pops and discards commands from the queue**
**File:** `MicService.kt` line 746â€“749 AND `apiController.js` line 88â€“91  
**Problem:** The `/api/heartbeat` endpoint calls `deviceStore.popQueuedCommands(deviceId)` and returns them in the response body. The Android client reads `commandsAvailable: true`, then calls `syncCommandsNow()` which calls `/api/sync` which **pops the same queue again** â€” but the commands are already gone! ### BUG-C5 Â· **Audio frame header format mismatch between APK and dashboard**
**File:** `MicService.kt` line 3151â€“3157 (writes), `useDashboard.ts` line 144â€“149 (reads)  
**Problem:** The APK writes a **4-byte audio header** `[0x4D, 0x4D, 0x01, codec]` then the PCM/MuLaw payload. The backend (`deviceController.js` line 306â€“309) prepends a **2-byte big-endian deviceId length + deviceId bytes** and forwards the whole thing. The dashboard reads offset 0 as `Uint16BE = deviceIdLen`, then at offset `2+deviceIdLen` it expects raw audio. So the 4-byte header is included in `data` sent to `useAudioPlayback.feedAudio`. The audio decoder must strip the 4 bytes before decoding PCM â€” if it doesn't, every frame is corrupt (header interpreted as audio samples â†’ static/noise at stream start).

### BUG-H1 Â· **Server-side double gain applied unconditionally â€” causing audio clipping**
**File:** `deviceController.js` line 296â€“303  
**Problem:** Server applies `serverGain = 2.5x` for both `lowNetwork` AND `farVoice` mode. The APK *already* applies a software gain of up to 9x in `applyFarVoiceGain()`. The combined gain can reach **22.5x**, which massively clips/distorts the audio. The dashboard hears either clipping static or a very distorted signal, not a clear loud voice.

**Fix:** Server gain should only compensate for MuLaw codec loss (when `streamCodec == "smart"`), not for far voice mode which is already boosted by the APK.

---

### BUG-H2 Â· **PCM stream sent even when WebRTC is active â€” wastes bandwidth in low-network mode**
**File:** `MicService.kt` line 2799: `if (!isWebRtcStreaming)`  
**Problem:** This guard correctly skips WS sends during WebRTC. BUT `applyFarVoiceGain()` (heavy FFT + gain pipeline) **always runs regardless** of `isWebRtcStreaming`. On low-end phones this wastes ~15-20ms of CPU per 40ms chunk, causing audio thread starvation and real WebRTC audio glitches.

**Fix:** Skip the entire `applyFarVoiceGain` call when `isWebRtcStreaming = true`.

---

### BUG-H3 Â· **Cellular network auto-triggers `lowNetworkMode = true`, which forces MuLaw in "smart" mode â€” breaking voice clarity on cellular**
**File:** `MicService.kt` line 373â€“377, and `chooseWsFallbackCodec()` line 3166â€“3174  
**Problem:** When the device switches to cellular, `onCapabilitiesChanged` sets `lowNetworkMode = true`. `chooseWsFallbackCodec()` in `smart` mode then returns `AUDIO_CODEC_MULAW_8K`. MuLaw at 8kHz has very poor voice quality â€” sounds muffled, robotic. The user wanted "clear voice on mobile data." The current logic actively degrades audio the moment mobile data is detected.

**Fix:** Keep `lowNetworkMode` only for explicit dashboard toggle or measured quality metrics (RTT/loss), not automatically on cellular detection.

---

### BUG-H4 Â· **`streamChunkSize` of 40ms at 16kHz is too large for low-network â€” causes perceptible gaps**
**File:** `MicService.kt` line 115  
```kotlin
private val streamChunkSize by lazy { ((sampleRate * 2) / 25).coerceAtLeast(1280) }
// = 32000/25 = 1280 bytes = 40ms
```
**Problem:** One chunk = 40ms of audio. On a bad connection, if a single WebSocket send stalls, the listener hears a 40ms gap (a clear voice "break"). Standard low-latency VoIP uses 20ms frames. The chunk size should be adaptive (20ms normal, 40ms only on very poor network as a packing optimization), not fixed at 40ms.

### BUG-H5 Â· **`isPhotoCaptureBusy` flag never reset if `captureJpegOnce` throws before setting `bytes`**
**File:** `MicService.kt` line 1986â€“2068  
**Problem:** `isPhotoCaptureBusy = true` is set before the `withTimeoutOrNull` block. If the coroutine is cancelled (service destroyed, scope cancelled), the `finally` block at line 2066 sets `isPhotoCaptureBusy = false`. BUT if `captureJpegOnce` throws a **coroutine cancellation exception** before the `try`, the `finally` of `serviceScope.launch` may not execute if the scope is already cancelled. Result: camera stays locked (busy=true), all future `take_photo` commands return `ACK:take_photo:busy`.

### BUG-H6 Â· **`sendCommandAck` uses `safeSend` which can trigger `onWsDisconnected` from a background thread â€” causing concurrent reconnect while a command is being processed**
**File:** `MicService.kt` line 704â€“706  
**Problem:** `safeSend` catches exceptions and launches `serviceScope.launch { onWsDisconnected(...) }`. If this happens inside `handleServerJsonCommand` (e.g., command_ack for `take_photo`), you get: command handling still running + reconnect tearing down audio + new WS connecting = race condition that can leave `isCapturing=true` but `audioRecord=null`.

### BUG-H7 Â· **`startCameraLiveStream` uses `CountDownLatch.await` on a coroutine IO thread â€” blocks the thread pool**
**File:** `MicService.kt` line 2437â€“2452  
**Problem:** `CountDownLatch(1).await(3, TimeUnit.SECONDS)` is a **blocking call** on a coroutine `IO` dispatcher thread. Kotlin coroutines explicitly warn against this. Under load (camera + audio + reconnect), this can exhaust the `IO` thread pool, causing audio send coroutines to be queued (delayed), resulting in audio stuttering or total silence for 3 seconds.

**Fix:** Use `suspendCancellableCoroutine` (already used correctly in `captureJpegOnce` camera open) for session creation here too.

### BUG-H8 Â· **`scheduleWebSocketReconnect` acquires `reconnectMutex` inside a new coroutine launched from `forceRestart=true` â€” but doesn't wait for the previous lock holder to release**
**File:** `MicService.kt` line 818â€“860  
**Problem:** When `forceRestart=true`, the old `wsReconnectJob` is cancelled. If the old job held `reconnectMutex.withLock`, cancelling it throws `CancellationException` **inside the lock**, which releases the lock automatically (Kotlin Mutex is fair). But the new job then immediately tries to acquire the lock and may race with the old job's `CancellationException` cleanup, causing a very brief window where `connectWebSocket()` is called twice.

### BUG-H9 Â· **Backend `startStreamRecovery` sends `start_stream` every 25s to ALL subscribed devices â€” causes audio floods on reconnect**
**File:** `dashboardController.js` line 329â€“357  
**Problem:** If `lastAudioChunkAt` is stale (e.e, for 25s), the backend sends `start_stream` again. But the device may already be streaming â€” `start_stream` causes `startAudioCapture()` again, which hits the double-start race (BUG-C2). On a weak network where audio chunks are infrequent, this fires every 25s repeatedly, continually restarting the audio capture loop.

### BUG-M6 Â· **`FcmMessageService.sendTokenToBackend` URL strips `/audio` suffix but not the `/<deviceId>` suffix**
**File:** `FcmMessageService.kt` line 63  
```kotlin
.replace(Regex("/audio(/.*)?$"), "")
```
The server URL stored in prefs is `wss://host/audio/<deviceId>`. This regex removes `/audio` and everything after, correctly. BUT `MicService.serverUrls` appends `/$deviceId` to the base: `wss://host/audio/$deviceId`. If the pref was saved including the deviceId path, the `replace` would remove `audio/deviceId` correctly. However if it was saved as just `wss://host/audio/`, the regex removes `/audio/` and the resulting URL is just `wss://host` â†’ FCM endpoint becomes `https://host/api/fcm-token`, which is correct. Edge case: if user manually types `wss://host` (no `/audio`), the regex doesn't match and the URL becomes `https://host/api/fcm-token` correctly. So this is **safe in practice but fragile** â€” a URL like `wss://host/audio/extrapath` would break it.

### BUG-L7 Â· **Backend `heartbeat` endpoint pops commands from the queue AND returns them in the response, but the APK ignores the `commands` field in the heartbeat response**
**File:** `apiController.js` line 90, `MicService.kt` line 744â€“750
